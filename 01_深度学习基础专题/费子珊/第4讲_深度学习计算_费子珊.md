## 5.1 层和块

### 5.1.1 核心概念：块（Block）

**块**是神经网络的基本抽象单元，可以描述：
- 单个层（如全连接层 `nn.Linear`）
- 由多个层组成的组件（如"一个残差块"）
- 整个模型本身（如整个 `MLP` 网络）

块的使用是**递归**的：可以将一些块组合成更大的组件，通过定义代码按需生成任意复杂度的块。

**块的分层结构（以ResNet为例）：**

整个模型是一个大块，包含多个阶段（stage）作为子块，每个阶段又包含多个残差块（residual block），而每个残差块内部包含卷积层、BatchNorm层、ReLU激活层等。这种嵌套结构可以用一棵树来描述：

- **第1层（根）**：整个模型
    - **第2层**：Stage 1（层组子块）
        - **第3层**：ResBlock 1, ResBlock 2, ...（残差块子块）
            - **第4层**：Conv2d, BatchNorm, ReLU（单个层/叶子节点）
    - **第2层**：Stage 2, Stage 3, Stage 4（同上结构）

> **补充说明**：这种分层设计借鉴了软件工程中的**组合模式（Composite Pattern）**。块既可以作为叶子节点（单个层），也可以作为组合节点（包含多个子块），这使得网络结构的设计非常灵活。在计算机视觉中的ResNet-152架构中，数百层由"层组"的重复模式组成；在NLP中，Transformer的编码器也是由多个相同的块堆叠而成。

### 5.1.2 块的基本功能

每个块必须提供以下基本功能：

| 功能 | 说明 | 实现方式 |
|------|------|----------|
| ① 接收输入 | 将输入数据作为前向传播函数的参数 | `forward(self, X)` |
| ② 生成输出 | 通过前向传播产生输出（形状可能与输入不同） | `return output` |
| ③ 计算梯度 | 计算输出关于输入的梯度 | 自动微分系统自动完成 |
| ④ 存储参数 | 存储和访问前向传播计算所需的参数 | `self.param = nn.Parameter(...)` |
| ⑤ 初始化参数 | 根据需要初始化模型参数 | `__init__` 构造函数 + 延后初始化 |

从编程角度来看，块由**类（class）**表示，其子类必须定义：
- `__init__`：构造函数，声明子模块和参数
- `forward`：前向传播函数，定义计算逻辑

反向传播由**自动微分**系统自动处理，不需要手动编写反向传播函数（这与早期框架如Caffe形成了鲜明对比，彼时需要手动推导和编写每个层的反向传播）。

### 5.1.3 `nn.Sequential` 回顾

```python
import torch
from torch import nn
from torch.nn import functional as F

net = nn.Sequential(nn.Linear(20, 256), nn.ReLU(), nn.Linear(256, 10))

X = torch.rand(2, 20)
net(X)
```

**`nn.Sequential` 的数据流（按执行顺序）：**

| 步骤 | 经过的层 | 输入形状 | 输出形状 | 参数情况 |
|------|----------|----------|----------|----------|
| 1 | `Linear(20→256)` | (2, 20) | (2, 256) | 权重 256×20 + 偏置 256（可训练） |
| 2 | `ReLU()` | (2, 256) | (2, 256) | 无参数，逐元素操作 |
| 3 | `Linear(256→10)` | (2, 256) | (2, 10) | 权重 10×256 + 偏置 10（可训练） |

`nn.Sequential` 是一种特殊的 `Module`，维护了一个由 `Module` 组成的有序列表。调用 `net(X)` 实际上是 `net.__call__(X)` 的简写。`__call__` 方法内部会先做一些准备工作（如设置训练/评估模式），然后再调用 `forward` 方法。

> **补充说明**：`net.__call__(X)` 不仅调用 `forward`，还负责钩子（hooks）的触发和一些内部状态的维护。这是为什么我们应该调用 `net(X)` 而不是直接调用 `net.forward(X)` 的原因——直接调用 `forward` 会跳过这些重要的前置和后置处理。

### 5.1.4 自定义块

通过继承 `nn.Module` 实现自定义块：

```python
class MLP(nn.Module):
    def __init__(self):
        super().__init__()                      # 调用父类构造函数
        self.hidden = nn.Linear(20, 256)        # 隐藏层
        self.out = nn.Linear(256, 10)           # 输出层

    def forward(self, X):
        return self.out(F.relu(self.hidden(X)))

net = MLP()
net(X)
```

**`nn.Module` 父类提供的核心属性和方法：**

| 类别 | 名称 | 类型 | 说明 |
|------|------|------|------|
| 属性 | `_modules` | OrderedDict | 存储所有子模块，键为模块名 |
| 属性 | `_parameters` | OrderedDict | 存储所有 `nn.Parameter` |
| 属性 | `_buffers` | OrderedDict | 存储非参数的持久化张量（如BatchNorm的running_mean） |
| 属性 | `training` | bool | 标记当前是训练模式还是评估模式 |
| 方法 | `__call__()` | — | 包装 `forward()`，前后做预处理和后处理 |
| 方法 | `forward()` | — | 子类必须重写的前向传播逻辑 |
| 方法 | `parameters()` | 生成器 | 递归返回所有 `nn.Parameter`（含子模块） |
| 方法 | `named_parameters()` | 生成器 | 同上，但返回 `(名称, 参数)` 元组 |
| 方法 | `to(device)` | — | 递归移动模块及其参数到指定设备 |
| 方法 | `train()` | — | 设置训练模式（`training=True`） |
| 方法 | `eval()` | — | 设置评估模式（`training=False`） |
| 方法 | `apply(fn)` | — | 深度优先递归对所有子模块执行函数 `fn` |
| 方法 | `state_dict()` | OrderedDict | 返回所有参数的名称→值映射（用于保存） |
| 方法 | `load_state_dict()` | — | 从字典加载参数（用于恢复） |
| 方法 | `add_module(name, module)` | — | 手动按名称注册子模块 |

**关键细节：**
- `super().__init__()` 调用父类构造函数，省去模板代码。这一步**必不可少**——如果忘记调用，父类的初始化逻辑不会执行，`_modules` 等核心属性不会被创建
- 层作为实例变量（`self.hidden`, `self.out`），`nn.Module` 的 `__setattr__` 方法会自动将 `nn.Module` 实例注册到 `_modules` 字典中
- 不同实例学到不同的模型参数（每个实例有自己的 `weight` 和 `bias` 副本）
- 反向传播和参数初始化由系统自动生成

> **补充说明**：`__setattr__` 的魔法：当你写 `self.hidden = nn.Linear(20, 256)` 时，Python 会调用 `nn.Module.__setattr__`，这个方法会检测到赋值的对象是一个 `nn.Module` 实例，然后自动将它注册到 `self._modules['hidden']` 中。这就是为什么后续可以通过 `net.named_parameters()` 递归遍历所有子模块的参数。

### 5.1.5 自定义顺序块（MySequential）

自己实现一个简化版 `Sequential` 来理解其工作原理：

```python
class MySequential(nn.Module):
    def __init__(self, *args):
        super().__init__()
        for idx, module in enumerate(args):
            self._modules[str(idx)] = module  # 存入OrderedDict

    def forward(self, X):
        for block in self._modules.values():
            X = block(X)
        return X

net = MySequential(nn.Linear(20, 256), nn.ReLU(), nn.Linear(256, 10))
net(X)
```

**为什么使用 `_modules` 而非 Python 列表？**

| 对比维度 | `_modules` (OrderedDict) | Python 列表 |
|----------|--------------------------|-------------|
| 遍历顺序 | 保证按添加顺序 | 也保证按添加顺序 |
| 参数发现 | 框架自动递归查找并初始化 | 框架**找不到**列表中的参数 |
| 设备迁移 | `net.to(device)` 自动迁移子模块 | 子模块被忽略，停留在原设备 |
| 序列化 | `state_dict()` 自动收集子模块参数 | 子模块参数丢失 |
| 命名管理 | 每个子模块有唯一名称 | 只有数字索引 |

> **补充说明**：使用 Python 普通列表（`self.modules_list = [...]`）的第一个后果就是：`net.to('cuda')` 不会把列表中的模块搬到GPU上，导致"部分模型在GPU、部分在CPU"的诡异bug。第二个后果是 `net.state_dict()` 会漏掉列表中的参数。这正是 `_modules`、`_parameters` 和 `_buffers` 这三个特殊 OrderedDict 存在的意义。

### 5.1.6 在前向传播中执行代码

不是所有架构都是简单的顺序架构。需要更强灵活性时可以定义自己的块，在前向传播中执行：

- **Python 控制流**（while 循环、if 语句等）—— 只要张量操作保持可导即可
- **任意数学运算** —— `torch.mm`、`torch.sum` 等都是可导的
- **常数参数**（不参与梯度更新的参数）—— 设置 `requires_grad=False`

```python
class FixedHiddenMLP(nn.Module):
    def __init__(self):
        super().__init__()
        # 不计算梯度的随机权重参数，训练期间保持不变
        self.rand_weight = torch.rand((20, 20), requires_grad=False)
        self.linear = nn.Linear(20, 20)

    def forward(self, X):
        X = self.linear(X)
        # 使用常量参数和ReLU
        X = F.relu(torch.mm(X, self.rand_weight) + 1)
        # 复用全连接层（参数共享）
        X = self.linear(X)
        # 控制流
        while X.abs().sum() > 1:
            X /= 2
        return X.sum()
```

**`FixedHiddenMLP` 前向传播执行步骤：**

| 步骤 | 操作 | 使用的参数 | 说明 |
|------|------|-----------|------|
| 1 | `self.linear(X)` | 可训练参数 W₁, b₁ | 第1次调用Linear层 |
| 2 | `X @ rand_weight + 1` | `rand_weight`（常量，`requires_grad=False`） | 固定权重的矩阵乘法 |
| 3 | `F.relu(...)` | 无 | ReLU激活 |
| 4 | `self.linear(X)` | 可训练参数 W₁, b₁（同一组） | 第2次调用同一Linear层，参数共享 |
| 5 | `while L1>1: X/=2` | 无 | Python控制流，动态迭代直到条件满足 |
| 6 | `X.sum()` | 无 | 归约为标量输出 |

> **补充说明**：这里的 `while` 循环是一个教学示例，实际应用中不太常见。但它展示了关键点：**只要计算图中的每个操作都是可导的（或者被标记为 `requires_grad=False`），autograd就能自动构建反向传播路径**。控制流（如 while、if）在计算图层面表现为动态图结构——每次前向传播时都会"重新录制"计算图。

### 5.1.7 块的混合搭配

可以灵活嵌套组合各种块：

```python
class NestMLP(nn.Module):
    def __init__(self):
        super().__init__()
        self.net = nn.Sequential(nn.Linear(20, 64), nn.ReLU(),
                                 nn.Linear(64, 32), nn.ReLU())
        self.linear = nn.Linear(32, 16)

    def forward(self, X):
        return self.linear(self.net(X))

chimera = nn.Sequential(NestMLP(), nn.Linear(16, 20), FixedHiddenMLP())
chimera(X)
```

**`chimera` 组合模型的结构拆解：**

| 索引 | 组件 | 内部结构 | 参数情况 |
|------|------|----------|----------|
| `[0]` | `NestMLP()` | `net`: Sequential(Linear(20→64) + ReLU + Linear(64→32) + ReLU) → `linear`: Linear(32→16) | 多组可训练参数 |
| `[1]` | `Linear(16→20)` | 单个全连接层 | 权重20×16 + 偏置20 |
| `[2]` | `FixedHiddenMLP()` | `rand_weight`(常量) + `linear`(Linear 20→20) | 一组可训练参数 + 一个常量张量 |

### 5.1.8 关于效率

读者可能会担心操作效率——高性能深度学习库中进行了字典查找、Python代码执行等。Python的全局解释器锁（GIL）是众所周知的瓶颈。在深度学习中，速度极快的GPU可能要等CPU运行Python代码后才能执行另一个作业。

**解决方式**：深度学习框架的前向传播在实际执行时，Python只作为"指挥官"——关键的数值计算全部由**C++/CUDA后端**执行。Python的 `forward` 方法只是在**构建计算图**，当实际输入到达时，大批量的矩阵运算在C++空间中异步执行，绕过GIL。

### 5.1.9 小结

| 要点 | 说明 |
|------|------|
| 块的组成 | 一个块可以由许多层组成，一个块也可以由许多块组成（递归组合） |
| 块的灵活性 | 块可以包含任意代码（控制流、数学运算、常量参数等） |
| 块的职责 | 负责参数初始化、反向传播、设备迁移等内部处理 |
| 顺序连接 | `Sequential` 块处理层和块的顺序连接 |
| 内部机制 | `_modules`、`_parameters`、`_buffers` 是框架管理组件的核心字典 |

## 5.2 参数管理

### 5.2.1 参数访问

#### 目标参数

通过 `Sequential` 定义的模型可以通过**索引**访问任意层，每层的参数在其属性中：

```python
import torch
from torch import nn

net = nn.Sequential(nn.Linear(4, 8), nn.ReLU(), nn.Linear(8, 1))
X = torch.rand(size=(2, 4))
net(X)

# 查看第二个全连接层的参数
print(net[2].state_dict())
# 输出: OrderedDict([('weight', tensor([[-0.0427, -0.2939, ...]])),
#                    ('bias', tensor([0.0887]))])
```

**`nn.Parameter` 对象包含了哪些信息？**

| 属性 | 类型 | 说明 |
|------|------|------|
| `.data` | Tensor | 参数的实际数值 |
| `.grad` | Tensor | 参数的梯度（反向传播后才有值，初始为None） |
| `.requires_grad` | bool | 是否需要计算梯度，默认为 `True`（即参与训练） |
| `.device` | device | 参数所在的设备（CPU/GPU） |
| `.dtype` | dtype | 参数的数据类型（float32等） |
| `.shape` | Size | 参数的形状 |

**`nn.Parameter` 与普通 Tensor 的关键区别**：
- `nn.Parameter` 会自动被注册到所在 Module 的 `_parameters` 字典中
- 普通 Tensor 即使被赋值给模块属性，也不会被 `model.parameters()` 发现
- 因此，优化器（如 `torch.optim.SGD`）只会更新 `nn.Parameter`，不会更新普通 Tensor

每个参数都表示为 `Parameter` 类的一个实例，包含值、梯度和额外信息：

```python
print(type(net[2].bias))          # <class 'torch.nn.parameter.Parameter'>
print(net[2].bias)                # Parameter containing: tensor([0.0887], requires_grad=True)
print(net[2].bias.data)           # tensor([0.0887])
print(net[2].weight.grad == None) # True（未调用反向传播时梯度为None）
```

> **补充说明**：`nn.Parameter` 是 `torch.Tensor` 的子类，但它有一个特殊行为：当被赋值给 `nn.Module` 的属性时，会自动注册到模块的 `_parameters` 字典中。这就是 `model.parameters()` 能遍历到所有参数的原理。

#### 一次性访问所有参数

```python
# 第一层的参数
print(*[(name, param.shape) for name, param in net[0].named_parameters()])
# 输出: ('weight', torch.Size([8, 4])) ('bias', torch.Size([8]))

# 整个网络的参数（带前缀标识）
print(*[(name, param.shape) for name, param in net.named_parameters()])
# 输出: ('0.weight', torch.Size([8, 4])), ('0.bias', torch.Size([8])),
#       ('2.weight', torch.Size([1, 8])), ('2.bias', torch.Size([1]))
```

参数命名规则：`{模块名}.{参数名}`，例如 `0` 是第一个模块的名字（Sequential按数字索引），`weight` 是该模块内部的参数名。

**三种访问参数方式的对比：**

| 方法 | 返回值 | 包含梯度？ | 使用场景 |
|------|--------|-----------|----------|
| `net.parameters()` | 生成器，每次产生一个 `nn.Parameter` | ✅ 是 | 传给优化器 |
| `net.named_parameters()` | 生成器，每次产生 `(name, param)` | ✅ 是 | 调试、选择性操作 |
| `net.state_dict()` | OrderedDict，键为名字、值为普通Tensor | ❌ 不含梯度信息 | 保存/加载模型 |

通过 `state_dict()` 访问参数：

```python
net.state_dict()['2.bias'].data  # tensor([0.0887])
```

#### 从嵌套块收集参数

构建嵌套网络，参数命名约定体现层级结构：

```python
def block1():
    return nn.Sequential(nn.Linear(4, 8), nn.ReLU(),
                         nn.Linear(8, 4), nn.ReLU())

def block2():
    net = nn.Sequential()
    for i in range(4):
        net.add_module(f'block {i}', block1())
    return net

rgnet = nn.Sequential(block2(), nn.Linear(4, 1))
```

**嵌套网络 `rgnet` 的层级结构（按访问路径展开）：**

| 索引路径 | 模块类型 | 参数路径示例 |
|----------|----------|-------------|
| `rgnet` | Sequential（最外层） | — |
| `rgnet[0]` | block2（Sequential，含4个子块） | — |
| `rgnet[0]['block 0']` → `rgnet[0][0]` | block1（Sequential） | — |
| `rgnet[0][0][0]` | Linear(4→8) | `0.block 0.0.weight` / `0.block 0.0.bias` |
| `rgnet[0][0][1]` | ReLU() | 无参数 |
| `rgnet[0][0][2]` | Linear(8→4) | `0.block 0.2.weight` / `0.block 0.2.bias` |
| `rgnet[0][0][3]` | ReLU() | 无参数 |
| `rgnet[0]['block 1']` ~ `['block 3']` | 同上结构 | 同上，分别名为 `block 1` ~ `block 3` |
| `rgnet[1]` | Linear(4→1) | `1.weight` / `1.bias` |

分层嵌套结构可通过索引链访问：

```python
rgnet[0][1][0].bias.data  # 第一个主块中第二个子块的第一层偏置
```

### 5.2.2 参数初始化

良好的初始化对数值稳定性至关重要。PyTorch 的 `nn.init` 模块提供了多种预置初始化方法。

**`apply` 方法的工作原理（深度优先递归遍历）：**

`net.apply(fn)` 的执行顺序：
1. 先对 `net` 自身调用 `fn(net)`
2. 递归进入 `net._modules` 中的每个子模块
3. 对子模块本身调用 `fn(子模块)`
4. 继续递归，直到叶子节点（如单个 Linear 层）
5. 对每个叶子层调用 `fn(Linear层)`
6. 返回 `net`，便于链式调用

> **补充说明**：`apply` 会对模块树中的每一个节点（包括容器模块和叶子层）都执行一次传入的函数。所以初始化函数中通常需要用 `if type(m) == nn.Linear` 做类型检查，否则会对 ReLU 等无参数层也尝试初始化。

#### 内置初始化

**高斯初始化**（标准差0.01，偏置为0）：

```python
def init_normal(m):
    if type(m) == nn.Linear:
        nn.init.normal_(m.weight, mean=0, std=0.01)
        nn.init.zeros_(m.bias)

net.apply(init_normal)
```

**常数初始化**（所有权重为1，偏置为0）：

```python
def init_constant(m):
    if type(m) == nn.Linear:
        nn.init.constant_(m.weight, 1)
        nn.init.zeros_(m.bias)

net.apply(init_constant)
```

**常用初始化方法速查表**：

| 初始化方法 | PyTorch 函数 | 适用场景 | 原理 |
|------------|-------------|----------|------|
| 正态分布 | `nn.init.normal_` | 通用场景 | 从正态分布 N(mean, std) 采样 |
| 均匀分布 | `nn.init.uniform_` | 通用场景 | 从均匀分布 U(a, b) 采样 |
| Xavier (Glorot) 均匀 | `nn.init.xavier_uniform_` | tanh/sigmoid激活 | 保持每层输入输出方差一致，范围由输入输出维度决定 |
| Xavier (Glorot) 正态 | `nn.init.xavier_normal_` | tanh/sigmoid激活 | 同上，但从正态分布采样 |
| Kaiming (He) 均匀 | `nn.init.kaiming_uniform_` | ReLU激活 ⭐推荐 | 为ReLU修正：方差放大2倍补偿被"砍掉"一半的激活值 |
| Kaiming (He) 正态 | `nn.init.kaiming_normal_` | ReLU及其变体 | 同上，但从正态分布采样 |
| 常数 | `nn.init.constant_` | 偏置初始化 | 全部设为指定常数 |
| 全零 | `nn.init.zeros_` | 偏置初始化 | 全部设为0 |
| 全一 | `nn.init.ones_` | 特殊场景 | 全部设为1 |
| 正交初始化 | `nn.init.orthogonal_` | RNN/LSTM | 权重矩阵为正交矩阵，有助于梯度稳定传播 |

> **补充说明**：**Xavier初始化** 的目标是使每一层输出的方差等于输入的方差，防止信号在前向传播中逐层放大或缩小，适用于tanh和sigmoid这类对称激活函数。**Kaiming初始化**（也叫He初始化）专门为ReLU设计——因为ReLU会"砍掉"一半的激活值，所以权重方差需要相应放大两倍来补偿。对于现代网络（大量使用ReLU/BatchNorm），He初始化通常是首选。

**对不同块应用不同初始化方法**：

```python
def init_xavier(m):
    if type(m) == nn.Linear:
        nn.init.xavier_uniform_(m.weight)    # Xavier初始化

def init_42(m):
    if type(m) == nn.Linear:
        nn.init.constant_(m.weight, 42)      # 常数42

net[0].apply(init_xavier)  # 第一层用Xavier
net[2].apply(init_42)      # 第三层用常数42
```

#### 自定义初始化

当框架未提供所需初始化方法时，可实现自定义初始化。例如实现如下分布：

$$w \sim \begin{cases} U(5, 10) & \text{概率 } \frac{1}{4} \\ 0 & \text{概率 } \frac{1}{2} \\ U(-10, -5) & \text{概率 } \frac{1}{4} \end{cases}$$

**自定义初始化的实现逻辑（两步走）：**

| 步骤 | 操作 | 效果 |
|------|------|------|
| 第1步 | `nn.init.uniform_(m.weight, -10, 10)` | 所有权重被随机初始化为 U(-10, 10)，值均匀覆盖整个 [-10, 10] 区间 |
| 第2步 | `m.weight.data *= m.weight.data.abs() >= 5` | `abs(w) >= 5` 的保留原值（|w| ∈ [5,10] 占1/2），`abs(w) < 5` 的被置零（占1/2）。保留的值中正负各半，各占总体的 1/4 |

最终效果完全匹配目标分布：50%的权重为0，25%在 [5,10] 区间，25%在 [-10,-5] 区间。

```python
def my_init(m):
    if type(m) == nn.Linear:
        nn.init.uniform_(m.weight, -10, 10)
        m.weight.data *= m.weight.data.abs() >= 5

net.apply(my_init)
```

也可以直接设置参数值：

```python
net[0].weight.data[:] += 1
net[0].weight.data[0, 0] = 42
```

> **补充说明**：直接操作 `.data` 时需要谨慎——这会绕过 autograd 的跟踪。只应在初始化阶段使用 `.data`。一旦训练开始，应通过优化器来更新参数。

### 5.2.3 参数绑定

在多个层间共享参数：

```python
shared = nn.Linear(8, 8)
net = nn.Sequential(nn.Linear(4, 8), nn.ReLU(),
                    shared, nn.ReLU(),
                    shared, nn.ReLU(),
                    nn.Linear(8, 1))

# 验证参数是否相同（不仅仅是值相同，而是同一对象）
print(net[2].weight.data[0] == net[4].weight.data[0])  # True
net[2].weight.data[0, 0] = 100
print(net[2].weight.data[0] == net[4].weight.data[0])  # True（同步变化）
```

**参数绑定的梯度处理：**

同一个参数 `shared.weight` 在计算图中出现了两次（`net[2]` 和 `net[4]`）：

| 阶段 | 发生了什么 |
|------|-----------|
| 前向传播 | 数据两次经过同一个 `shared` 层，每次使用相同的权重矩阵 W |
| 反向传播 | `shared` 从第1次使用收到梯度 ∂L/∂W₁，从第2次使用收到梯度 ∂L/∂W₂ |
| 梯度累加 | PyTorch的autograd引擎自动将两个梯度相加：**W的最终梯度 = ∂L/∂W₁ + ∂L/∂W₂** |
| 参数更新 | 优化器使用累加后的总梯度来更新 W |

**参数绑定的应用场景**：

| 场景 | 说明 | 例子 |
|------|------|------|
| 减少参数量 | 多个位置需要相同变换时共享权重 | Siamese网络中的孪生分支 |
| 权重绑定（Weight Tying） | 编码器和解码器共享词嵌入矩阵 | Transformer、Seq2Seq模型 |
| 等价约束 | 强制不同部分学习相同特征 | 自编码器中 encoder/decoder 权重互为转置 |

> **补充说明**：参数绑定的梯度累加性质来源于多元微积分中的**链式法则**。当一个参数在计算图中出现多次时，它从每条路径都收到一个梯度信号，这些信号会自动相加（PyTorch的autograd引擎会在反向传播时对同一参数的多个梯度进行累加）。这就是共享参数训练在数学上的本质。

### 5.2.4 小结

| 要点 | 说明 |
|------|------|
| 参数访问 | 可通过索引、`named_parameters()`、`state_dict()` 等多种方式访问 |
| 参数初始化 | 框架提供内置初始化方法（normal、xavier、constant、kaiming），也支持自定义 |
| 参数绑定 | 多个层可共享同一参数对象，梯度在反向传播时会累加 |
| `apply` 方法 | 可对模型的所有模块递归应用函数（深度优先遍历） |

## 5.3 延后初始化

### 5.3.1 什么是延后初始化

延后初始化（Deferred Initialization）是指：深度学习框架在**定义网络架构时不立即初始化参数**，而是等到**数据第一次通过网络传递时**，才动态推断每个层的大小并初始化。

**两种方式对比：**

| 对比维度 | 传统方式（手动指定维度） | 延后初始化方式 |
|----------|------------------------|---------------|
| 定义时 | 必须明确知道：输入维度 → 每层输出维度 → ... → 最终输出维度 | 可以"先不填"输入维度 |
| 修改网络时 | 必须手动更新所有受影响的维度声明 | 只管加减层，维度自动适配 |
| 易出错程度 | 维度不匹配导致运行时错误，排查麻烦 | 框架自动推断，不会出现维度声明错误 |
| 灵活性 | 每次换输入尺寸都要改代码 | 输入尺寸随意更换 |

**好处**：
- 无需手动指定输入维度（减少模板代码）
- 添加层时不需指定前一层的输出维度（灵活组合）
- 在初始化参数时不需要事先知道网络应该包含多少参数
- 使用卷积网络时更加方便（输入图像分辨率影响每层维度，手动计算各层的输出尺寸非常繁琐）
- 大大简化模型的**定义和修改**

> **补充说明**：在使用卷积网络时，输入图像的分辨率（如224×224 vs 299×299）会通过卷积核大小、步长、填充的层层作用，影响每一层特征图的空间尺寸和通道数。手动为每一层计算输出维度不仅枯燥，而且每次修改架构或输入尺寸都要重新计算。延后初始化让框架自动完成这项工作。

### 5.3.2 工作过程

延后初始化分为三个阶段，以 `MLP()` 为例：

**阶段1：定义网络（不初始化）**

`net = MLP()` 执行后：
- 网络对象被创建
- `_modules` 字典中有子模块，但 `_parameters` 为空
- 各层权重矩阵的形状**未知**——比如第一层 Linear 还不知道输入维度是几
- 状态比喻：建筑图纸画好了，但材料还没准备

**阶段2：数据首次通过网络（触发初始化）**

假设传入数据 `X.shape = (2, 20)`，即 `net(X)` 执行时：

| 步骤 | 框架做了什么 |
|------|-------------|
| ① 到达第一层 Linear | 发现输入维度 = 20，确定 `weight.shape = (256, 20)`，分配内存并用默认方法初始化参数 |
| ② 到达隐藏层 | 上一层输出维度 = 256 自然成为本层输入，确定本层参数形状，分配并初始化 |
| ③ 到达输出层 | 同样根据输入维度确定最后一层的参数形状，分配并初始化 |
| 完成 | 所有参数形状已知，初始值已填充 |

**阶段3：后续使用（正常执行）**

之后的每次前向传播，所有参数形状已知，直接执行计算和反向传播，不再有初始化行为。

**关键点**：只有第一层需要"等待"输入来确定形状。一旦第一层确定了输出尺寸，后续层的输入维度就都自然确定了。但框架为了统一处理，仍然是按顺序逐层初始化的。

> **补充说明**：在PyTorch中，`nn.Linear(20, 256)` 其实并不是真正的"延后初始化"——它明确指定了输入和输出维度。PyTorch的延后初始化主要出现在 `nn.LazyLinear(256)` 这类Lazy模块中。但书中将这个概念作为理解框架如何推断参数的桥梁来介绍，其思想在不同框架间是相通的（如MXNet的延后初始化更为彻底）。

### 5.3.3 小结

| 要点 | 说明 |
|------|------|
| 自动推断 | 框架自动推断参数形状，简化模型架构修改 |
| 触发时机 | 通过模型传递数据后，框架才真正初始化参数 |
| 避免错误 | 避免了一些因手动指定维度而产生的常见错误 |
| 实现原理 | 第一层从输入数据获知维度，后续层从前一层输出推断 |

## 5.4 自定义层

### 5.4.1 不带参数的层

继承基础层类 `nn.Module`，只需实现 `forward` 方法：

```python
import torch
import torch.nn.functional as F
from torch import nn

class CenteredLayer(nn.Module):
    def __init__(self):
        super().__init__()

    def forward(self, X):
        return X - X.mean()  # 从输入中减去均值

# 测试
layer = CenteredLayer()
layer(torch.FloatTensor([1, 2, 3, 4, 5]))  # tensor([-2., -1.,  0.,  1.,  2.])
```

> **补充说明**：这个层看起来简单，但它实际上在做**零中心化（zero-centering）**——一种常见的数据预处理操作。将其封装为一个层意味着它可以被插入网络的任意位置，在前向传播中对中间特征做归一化。其反向传播梯度会自动计算：`∂(X - mean(X))/∂X = 1 - 1/n`（对每个元素）。

可以将其作为组件合并到复杂模型中：

```python
net = nn.Sequential(nn.Linear(8, 128), CenteredLayer())
Y = net(torch.rand(4, 8))
Y.mean()  # tensor(7.4506e-09) ≈ 0（浮点精度误差）
```

**不带参数层 vs 带参数层 的对比：**

| 对比维度 | 不带参数的层 | 带参数的层 |
|----------|-------------|-----------|
| `__init__` 中 | 无需声明 `nn.Parameter` | 必须用 `nn.Parameter()` 包装张量 |
| 状态 | 无状态，仅做纯数学变换 | 有可训练状态，会随训练变化 |
| 训练中 | 行为不变 | 参数被优化器不断更新 |
| 典型例子 | ReLU, Dropout, MaxPool, Flatten | Linear, Conv2d, BatchNorm, Embedding |
| 反向传播 | 自动计算梯度（如 ∂ReLU/∂x） | 除激活梯度外还需计算参数梯度 |
| `named_parameters()` | 不包含 | 包含 |

### 5.4.2 带参数的层

使用 `nn.Parameter` 创建可通过训练调整的参数：

```python
class MyLinear(nn.Module):
    def __init__(self, in_units, units):
        super().__init__()
        self.weight = nn.Parameter(torch.randn(in_units, units))
        self.bias = nn.Parameter(torch.randn(units,))

    def forward(self, X):
        linear = torch.matmul(X, self.weight.data) + self.bias.data
        return F.relu(linear)

# 实例化并访问参数
linear = MyLinear(5, 3)
linear.weight  # Parameter containing: tensor(...)

# 直接执行前向传播
linear(torch.rand(2, 5))

# 像内置层一样使用
net = nn.Sequential(MyLinear(64, 8), MyLinear(8, 1))
net(torch.rand(2, 64))
```

**`nn.Parameter` 的优势：**

| 优势 | 说明 |
|------|------|
| 自动注册 | 自动被 `model.parameters()` 和 `model.state_dict()` 发现 |
| 设备迁移 | `model.to(device)` 自动将其移动到目标设备 |
| 序列化 | `torch.save` 自动保存，`load_state_dict` 自动恢复 |
| 类型转换 | `model.float()` / `model.double()` 自动转换精度 |
| 梯度管理 | 自动参与反向传播，优化器自动更新 |

> **补充说明**：注意上面代码中的 `self.weight.data` 和 `self.bias.data`。这里使用 `.data` 会得到一个不带梯度追踪的普通Tensor，然后与输入进行矩阵乘法。在实际的 `nn.Linear` 源码中，使用的是 `F.linear(input, self.weight, self.bias)`，它直接使用带梯度的Parameter，以便autograd能追踪参数与输入之间的依赖关系来做反向传播。本书教程中使用 `.data` 是为了教学清晰。

### 5.4.3 小结

| 要点 | 说明 |
|------|------|
| 无参数层 | 继承 `nn.Module`，仅实现 `forward`，内部无 `nn.Parameter` |
| 有参数层 | 使用 `nn.Parameter` 包装张量，使框架自动管理参数生命周期 |
| 灵活性 | 自定义层可在任意环境和网络架构中调用 |
| 设计自由 | 可以定义行为与现有层完全不同的新层 |
| 与内置层等价 | 自定义层与内置层在使用方式上完全一致 |

## 5.5 读写文件

### 5.5.1 保存和加载张量

**`torch.save` 和 `torch.load` 的功能：**

| 函数 | 方向 | 支持的数据类型 |
|------|------|---------------|
| `torch.save(obj, path)` | 内存 → 磁盘 | 单个张量、张量列表、字典（str→Tensor）、任何可pickle的Python对象 |
| `torch.load(path)` | 磁盘 → 内存 | 恢复保存的Python对象（在PyTorch环境中） |

单个张量的读写：

```python
import torch
from torch import nn
from torch.nn import functional as F

# 保存
x = torch.arange(4)
torch.save(x, 'x-file')

# 加载
x2 = torch.load('x-file')
x2  # tensor([0, 1, 2, 3])
```

存储和读取张量列表：

```python
y = torch.zeros(4)
torch.save([x, y], 'x-files')
x2, y2 = torch.load('x-files')
```

存储和读取字典（从字符串映射到张量）：

```python
mydict = {'x': x, 'y': y}
torch.save(mydict, 'mydict')
mydict2 = torch.load('mydict')
```

> **补充说明**：`torch.save` 底层使用 Python 的 `pickle` 序列化机制，但它对张量做了优化（直接序列化底层存储而非Python对象包装）。这意味着保存的文件只能在PyTorch中加载，不能跨框架使用。如果需要跨框架，应该导出为ONNX、TorchScript等通用格式。

### 5.5.2 保存和加载模型参数

**重要注意**：保存的是模型**参数**，而不是整个模型。因为模型本身可以包含任意代码（if/while等），难以序列化。

**保存与加载的完整工作流：**

**保存阶段（4步）：**

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | 训练好模型 `net` | 经过完整训练，参数已收敛 |
| 2 | `params = net.state_dict()` | 提取 OrderedDict：`{'hidden.weight': tensor(...), 'hidden.bias': tensor(...), 'output.weight': tensor(...), 'output.bias': tensor(...)}` |
| 3 | `torch.save(params, 'mlp.params')` | 将参数字典序列化到磁盘文件 |
| 4 | 磁盘上得到文件 `mlp.params` | 该文件**只包含参数值**，不含任何架构信息 |

**加载阶段（4步）：**

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | `params = torch.load('mlp.params')` | 从磁盘文件反序列化，得到 OrderedDict |
| 2 | `clone = MLP()` | **用代码重建相同的架构**（这一步必须手动完成） |
| 3 | `clone.load_state_dict(params)` | 将参数字典中的值逐个赋值给 clone 的对应层 |
| 4 | `clone.eval()` | 切换到评估模式（关闭Dropout等训练特有的行为） |

**保存流程：**

```python
class MLP(nn.Module):
    def __init__(self):
        super().__init__()
        self.hidden = nn.Linear(20, 256)
        self.output = nn.Linear(256, 10)

    def forward(self, x):
        return self.output(F.relu(self.hidden(x)))

net = MLP()
X = torch.randn(size=(2, 20))
Y = net(X)

# 保存模型参数
torch.save(net.state_dict(), 'mlp.params')
```

**加载流程：**

```python
# 实例化模型备份（需用代码定义相同的架构）
clone = MLP()
# 从文件读取参数
clone.load_state_dict(torch.load('mlp.params'))
clone.eval()  # 设置为评估模式（关闭Dropout等）

# 验证参数一致性
Y_clone = clone(X)
Y_clone == Y  # tensor([[True, True, ...]])
```

> **补充说明**：`clone.eval()` 将模型设置为评估模式。这很重要——如果模型中有 Dropout 或 BatchNorm 层，它们在训练模式和评估模式下的行为是不同的。如果加载参数后不调用 `eval()`，在推理时 Dropout 仍会随机丢弃神经元，导致每次推理结果不同。

### 5.5.3 小结

| 要点 | 说明 |
|------|------|
| 张量读写 | `torch.save()` 和 `torch.load()` 用于保存和加载任意Python对象（张量/列表/字典） |
| 模型保存 | 通过 `state_dict()` 保存参数字典，不保存架构代码 |
| 模型恢复 | 需要先用代码重建相同架构，再用 `load_state_dict()` 加载参数 |
| 中断恢复 | 定期保存中间结果（checkpoint），防止服务器断电导致计算损失 |
| 评估模式 | 加载后调用 `eval()` 确保Dropout、BatchNorm等层切换到推理行为 |

## 5.6 GPU计算

### 5.6.1 计算设备

**CPU与GPU的架构差异：**

| 对比维度 | CPU | GPU |
|----------|-----|-----|
| 核心数量 | 少数（如4~16核） | 数千个CUDA核心 |
| 擅长的任务 | 逻辑控制、串行任务、低延迟响应 | 大规模并行计算、矩阵运算 |
| 内存 | 主机内存（大容量，如32GB+） | 显存（容量相对小，如8~24GB） |
| 延迟 | 低 | 较高（但吞吐量巨大） |
| 通信方式 | CPU↔GPU 通过 PCIe 总线（慢） | GPU↔GPU 通过 NVLink/NVSwitch（较快） |

**重要提示**：CPU和GPU之间的数据传输（通过PCIe总线）是**最大的性能瓶颈**。GPU内部计算极快，但如果频繁在CPU和GPU之间搬运数据，性能会急剧下降。

在 PyTorch 中，CPU 和 GPU 用 `torch.device` 表示：

```python
import torch
from torch import nn

torch.device('cpu')           # device(type='cpu')
torch.device('cuda')          # device(type='cuda')  ← 等价于 cuda:0
torch.device('cuda:1')        # device(type='cuda', index=1)
```

查询可用 GPU 数量：

```python
torch.cuda.device_count()     # 返回可用GPU数量
```

**方便的辅助函数**（当所需 GPU 不存在时回退到 CPU）：

```python
def try_gpu(i=0):
    """如果存在，则返回gpu(i)，否则返回cpu()"""
    if torch.cuda.device_count() >= i + 1:
        return torch.device(f'cuda:{i}')
    return torch.device('cpu')

def try_all_gpus():
    """返回所有可用的GPU，如果没有GPU，则返回[cpu(),]"""
    devices = [torch.device(f'cuda:{i}')
               for i in range(torch.cuda.device_count())]
    return devices if devices else [torch.device('cpu')]

try_gpu(), try_gpu(10), try_all_gpus()
# 结果: (device(type='cuda', index=0),
#        device(type='cpu'),          ← GPU 10不存在，自动回退CPU
#        [device(type='cuda', index=0), device(type='cuda', index=1)])
```

> **补充说明**：`try_gpu()` 和 `try_all_gpus()` 是很好的实践模式，让代码在有无GPU的环境下都能运行。在生产环境中，通常会配合环境变量或配置文件来指定使用的设备，而不是硬编码设备编号。

### 5.6.2 张量与GPU

#### 查询张量所在设备

```python
x = torch.tensor([1, 2, 3])
x.device  # device(type='cpu')  ← 默认在CPU上
```

#### 在GPU上创建张量

```python
X = torch.ones(2, 3, device=try_gpu())
X  # tensor([[1., 1., 1.], [1., 1., 1.]], device='cuda:0')

# 在第二个GPU上创建
Y = torch.rand(2, 3, device=try_gpu(1))
Y  # device='cuda:1'
```

> **补充说明**：在GPU上创建的张量只消耗该GPU的显存。可以使用 `nvidia-smi` 命令查看显存使用情况。需要确保不创建超过GPU显存限制的数据——一旦OOM（Out of Memory），程序会直接崩溃。

**⚠️ 关键规则：多个项进行操作时，它们必须在同一设备上。**

| 情况 | X所在设备 | Y所在设备 | X + Y 结果 |
|------|----------|----------|-----------|
| ✅ 正确 | GPU:0 | GPU:0 | 正常计算，结果在GPU:0 |
| ✅ 正确 | CPU | CPU | 正常计算，结果在CPU |
| ❌ 错误 | GPU:0 | GPU:1 | `RuntimeError: Expected all tensors to be on the same device` |
| ❌ 错误 | GPU:0 | CPU | `RuntimeError: Expected all tensors to be on the same device` |

#### 设备间复制数据

```python
Z = X.cuda(1)   # 将X从GPU:0复制到GPU:1
print(Y + Z)    # 现在Y和Z都在GPU:1上，可以计算

# 如果目标设备与当前设备相同，不会复制（幂等操作）
Z.cuda(1) is Z  # True — 已在GPU:1上，直接返回自身
```

**不同设备间数据传输的代价对比：**

| 传输方向 | 速度 | 原因 | 建议 |
|----------|------|------|------|
| CPU → GPU | 慢 | 通过PCIe总线，带宽远低于显存带宽 | 训练开始时一次性搬过去 |
| GPU → CPU | 慢 | 同上 | 训练结束后一次性搬回来 |
| GPU → GPU（同机） | 较快 | NVLink/NVSwitch直连，带宽高一个数量级 | 可直接使用 |
| GPU内部计算 | 极快 | 数千核心并行，显存带宽极高 | 尽量让数据留在GPU上 |

**经验法则**：一次大数据传输 + 一次大计算，远好于多次小数据传输 + 多次小计算。将数据传输开销分摊到大计算上。

### 5.6.3 神经网络与GPU

将模型参数移动到GPU：

```python
net = nn.Sequential(nn.Linear(3, 1))
net = net.to(device=try_gpu())

# 输入在GPU上时，模型自动在GPU上计算
net(X)  # device='cuda:0'

# 确认模型参数在同一GPU上
net[0].weight.data.device  # device(type='cuda', index=0)
```

**GPU训练的完整数据流（每个训练迭代）：**

| 步骤 | 操作 | 所在设备 | 说明 |
|------|------|----------|------|
| ① | 加载数据 | CPU | `DataLoader` 通常在CPU上产出batch |
| ② | 数据迁移 | CPU → GPU | `X = X.to('cuda')`；`y = y.to('cuda')` |
| ③ | 前向传播 | GPU | 模型已在GPU上，计算自动在GPU执行 |
| ④ | 计算损失 | GPU | loss函数在GPU上执行 |
| ⑤ | 反向传播 | GPU | `loss.backward()`，梯度在GPU上 |
| ⑥ | 参数更新 | GPU | `optimizer.step()`，参数原地更新 |
| ⑦ | （可选）日志 | GPU → CPU | `loss.item()` 触发隐式GPU→CPU传输！ |

> ⚠️ 步骤⑦经常被忽略，但它会**打破GPU流水线**，造成性能损失。详见下一节。

### 5.6.4 性能注意事项

| 原则 | 说明 | 反例 |
|------|------|------|
| 设备间数据传输比计算慢得多 | 尽量避免频繁的 CPU↔GPU 数据传输 | 每个batch都把整个数据从CPU搬到GPU再搬回来 |
| 多个小操作 < 一个大操作 | 批量处理，减少内核启动开销 | 用for循环逐个样本做推理而不是整个batch |
| 一次执行 > 散布操作 | 合并操作，减少同步点 | 散布的`item()`调用打断GPU流水线 |
| 避免隐式传输 | 打印/转NumPy会触发GPU→CPU复制 | 每一步都打印loss到控制台 |

**典型错误对比：**

**❌ 错误做法 — 每步都触发 GPU→CPU 传输：**
```python
for batch in dataloader:
    X, y = batch
    X = X.to('cuda')
    y = y.to('cuda')
    loss = model(X, y)
    print(loss.item())   # ← 触发 GPU→CPU 传输，阻塞GPU
    numpy_loss = loss.detach().cpu().numpy()  # ← 又一次传输
    ...
```

**✅ 正确做法 — 训练结束后一次性搬回CPU：**
```python
all_losses = []
for batch in dataloader:
    X, y = batch
    X = X.to('cuda')
    y = y.to('cuda')
    loss = model(X, y)
    all_losses.append(loss.detach())  # 保留在GPU上

# 训练结束后一次性搬回CPU
avg_loss = torch.stack(all_losses).mean().cpu().item()
```

> **补充说明**：`loss.item()` 看起来无害，但它会触发一次从GPU到CPU的数据传输，并由于Python的GIL，在数据传输期间GPU可能处于空闲状态。在大规模训练中，这种隐式的小传输累积起来会显著拖慢训练速度。

### 5.6.5 小结

| 要点 | 说明 |
|------|------|
| 计算设备 | 可指定CPU或GPU存储和计算，默认在主内存和CPU |
| 设备一致性 | 框架要求所有输入数据在同一设备上（CPU或GPU） |
| 性能陷阱 | 不经意地移动数据会显著降低性能（含隐式传输） |
| 多GPU | `cuda:0`、`cuda:1` 等表示不同GPU卡，`try_gpu()` 提供安全回退 |
| 模型迁移 | `net.to(device)` 递归将整个模型及其参数移到指定设备 |
| 批量处理 | GPU偏好大批量操作，小批量频繁操作效率低下 |
